# GitHub Flow 学習ガイド

GitHub CLI (`gh`) を活用したモダンな開発フローの手順です。

## 1. Issue の作成
まず、何を修正・追加するかを Issue として登録します。

```powershell
gh issue create --title "Issueのタイトル" --body "Issueの内容"
```

## 2. ブランチの作成と切り替え
作業用の新しいブランチを作成します。

```powershell
git checkout -b branch-name
```

## 3. ファイルの修正とコミット
通常通りファイルを修正し、コミットします。

```powershell
git add .
git commit -m "コミットメッセージ"
```

## 4. GitHub へプッシュ
作成したブランチを GitHub へ送信します。初回は `-u origin branch-name` を付けます。

```powershell
git push -u origin branch-name
```

## 5. プルリクエスト (PR) の作成
GitHub CLI を使って PR を作成します。

```powershell
gh pr create --title "PRのタイトル" --body "PRの詳細内容" --base main
```

## 6. PR のマージ
レビューが完了したら、PR をメインブランチにマージします。作業ブランチの削除も同時に行えます。

```powershell
gh pr merge --merge --delete-branch
```

## 各作業の意味合いとメリット

| ステップ | 作業 | 意味合い | メリット |
| :--- | :--- | :--- | :--- |
| **1. Issue** | 目的の宣言 | 「何をするか」を明確にする。 | 作業の漏れを防ぎ、後で振り返りやすくなる。 |
| **2. Branch** | 作業場の隔離 | 本番（main）から離れた「実験場」を作る。 | 失敗しても本番に影響が出ない。複数の作業を並行できる。 |
| **3. Commit/Push** | 履歴の保存 | 変更を記録し、サーバーにバックアップする。 | 「いつ、何をしたか」が残り、過去の状態に戻せるようになる。 |
| **4. PR (Pull Request)** | 変更の提案 | 「この変更を main に入れてもいい？」と伺いを立てる。 | コードの意図を説明し、自分や他人が確認（レビュー）する場になる。 |
| **5. Merge** | 変更の統合 | 承認された変更を本番（main）に合流させる。 | 正式にプロジェクトの一部として採用される。 |
| **6. Pull** | 最新化 | 他の場所での変更をローカルに取り込む。 | 常に最新の「正解」を手元に持っておける。 |

## なぜ「ちょっとした修正」でもブランチを作るのか？

結論から言うと、**「本番 (main) ブランチを常に壊さないため」**です。

1.  **安全性の確保**:
    修正中に「やっぱり別のやり方が良かった」となったとき、ブランチならその作業を捨てるだけで済みます。main で直接作業していると、元に戻すのが大変になることがあります。
2.  **歴史の透明性**:
    「Issue #1 のための修正」という単位で履歴がまとまるため、後から「なぜこの一行を変えたのか？」を調べるときに非常に助かります。
3.  **CI/CD (自動テスト) との相性**:
    多くの現場では、PR を作成した瞬間に自動でテストが走る仕組みになっています。ブランチを使うことで、バグを含んだコードが main に入る前に自動で検知できます。
4.  **並行作業**:
    ある修正の途中で急に別の優先度の高い修正（バグ修正など）が入ったとき、ブランチを分けていればスムーズに切り替えられます。

一人での開発でも、将来的に誰かと協力したり、プロジェクトが大きくなったりしたときに「ブランチを切る癖」がついていると強力な武器になります。
